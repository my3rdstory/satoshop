## 배경
- Render 테스트 서버에서 디스코드 앱 키를 교체했는데도 슬래시 명령 실행 시 "애플리케이션이 응답하지 않았어요"가 발생할 수 있는 상황이 있었습니다.
- 기존 인터랙션 처리는 `get_active_bot()` 1건만 사용해 서명을 검증했기 때문에, 활성 봇이 여러 개이거나 앱 교체 직후 설정 혼선이 있으면 서명 검증 실패 가능성이 있었습니다.

## 변경 사항
- 인터랙션 요청 본문(JSON)의 `application_id`를 먼저 읽고, 해당 앱 ID의 활성 봇 설정을 우선 조회하도록 변경했습니다.
- 앱 ID 기반 조회 실패 시에만 기존 활성 봇 조회를 fallback으로 사용하도록 보강했습니다.
- 서명 검증 실패 시 `bot_id`, `payload_app_id`, `bot_app_id`를 로그에 남겨 원인 추적이 가능하도록 했습니다.
- 인터랙션 처리 중 예외가 발생해도 500으로 끊기지 않도록 예외를 캡처하고, 에페메랄 오류 메시지를 즉시 응답하도록 변경했습니다.
- README에 앱 ID 기반 검증 동작과 다중 활성 봇 운영 시 주의사항을 추가했습니다.

## 영향
- 테스트/운영 봇이 혼재된 환경에서도 디스코드 payload의 앱 ID 기준으로 정확한 봇 키를 선택해 검증할 수 있습니다.
- 장애 발생 시 로그 단서가 늘어나 원인 파악 시간이 단축됩니다.
- 예외 상황에서도 디스코드 사용자에게 즉시 응답이 반환되어 "응답 없음" 체감이 줄어듭니다.

## 대장 수동 테스트 가이드
1. Render 테스트 서버 어드민에서 개발용 앱의 `application_id`, `public_key`, `token`을 저장합니다.
2. `uv run python manage.py sync_discord_commands --all-guilds`를 1회 실행합니다.
3. 디스코드 테스트 채널에서 `/사토샵_최근등록상품` 실행 후 응답 도착 여부를 확인합니다.
4. 실패 시 Render 로그에서 `디스코드 서명 검증 실패` 또는 `디스코드 인터랙션 처리 중 오류` 로그를 확인합니다.
